# 默认聊天模型唯一来源设计

## 背景

当前仓库已经支持在模型管理页手动设置“默认聊天模型”，但服务端运行时仍会在数据库未设置时回退到 `application.yml` 里的 `default-chat-model`。这会形成“后台配置一套、配置文件再兜底一套”的双事实来源，和“默认聊天模型应由模型管理模块手动维护”的产品约束冲突。

## 设计结论

- 默认聊天模型的唯一事实来源改为数据库 `ai_model.is_default_chat_model`
- 运行时只认后台手动设置的默认聊天模型，不再读取 `application.yml` 的聊天默认模型配置
- 全局默认聊天模型只能有一个，且只能是启用中的聊天模型
- 如果当前没有默认聊天模型，问答链路直接返回明确错误，提示管理员先到模型管理中设置

## 约束落点

### 后端

- `ModelService.resolveChatModelDescriptor(null)` 只从数据库解析默认聊天模型
- 当前默认聊天模型禁止直接删除
- 当前默认聊天模型禁止被改成禁用状态或改成非聊天用途
- 配置文件中继续保留提供方接入参数和默认向量模型配置，不再保留默认聊天模型配置

### 前端

- 模型管理页明确提示“默认聊天模型只能在本页手动设置，且必须保持唯一”
- 知识库表单中的“留空兜底”提示改成“使用模型管理中设置的默认聊天模型”
- 删除默认聊天模型时前端先给出阻断提示，减少无效请求

## 风险与取舍

- 新环境首次启动后，如果管理员还没有设置默认聊天模型，聊天相关链路会报错
- 这是有意取舍，目的是避免配置文件和后台配置不一致导致的运行歧义
- 向量模型仍允许配置文件兜底，避免本次变更扩大到 Embedding 默认策略
