# 失败任务重试设计说明

## 1. 目标

本次实现目标是在现有任务监控页基础上，补齐“失败任务重试”最小治理动作：

- 对 `FAILED` / `CANCELED` 任务提供重试入口
- 点击前进行确认
- 当前行进入 loading
- 调用任务重试接口
- 成功后刷新当前列表

本轮重点是把任务监控页从只读可观察，推进到最小可操作治理。

## 2. 方案选择

本次采用“任务列表内重试 + 确认提示 + 行级 loading”方案。

选择原因如下：

- 交互路径最短
- 可以明确定位到具体任务
- 风险可控，不需要为了当前能力提前引入任务详情页

## 3. 交互设计

在任务监控页表格“操作”列新增 `重试` 按钮。

显示规则：

- `FAILED`：显示
- `CANCELED`：显示
- 其他状态：不显示

交互流程如下：

1. 用户点击 `重试`
2. 弹出确认提示
3. 用户确认后，当前行按钮进入 loading
4. 调用 `POST /api/admin/tasks/{taskId}/retry`
5. 成功后提示“任务已重新投递”
6. 刷新当前列表，保留当前筛选条件和分页

## 4. 数据流设计

前端新增任务重试接口封装，对接 `POST /api/admin/tasks/{taskId}/retry`。

状态更新策略：

- 不在前端本地猜测任务状态
- 重试成功后直接刷新当前列表

## 5. 错误处理

错误处理规则如下：

- 普通失败：显示后端返回的错误信息
- 规则失败：例如后端拒绝重试，也直接显示明确提示
- 当前行 loading 结束后允许再次操作

## 6. 本轮边界

本轮包含：

- 任务列表重试按钮
- 操作确认
- 行级 loading
- 成功 / 失败提示
- 操作后刷新当前列表

本轮不包含：

- 任务详情页
- 失败原因详情弹窗
- 批量重试
- 自动轮询重试结果

## 7. 验证范围

本轮至少完成以下验证：

- 只有 `FAILED` / `CANCELED` 任务出现重试按钮
- 点击重试会先确认
- 确认后可调用接口并刷新当前列表
- 重试失败会显示明确提示
- 前端项目可通过 `npm run build`

## 8. 后续建议

在本轮完成后，建议继续按以下顺序推进：

1. 任务详情页
2. 失败原因展开查看
3. 任务与文档详情联动
