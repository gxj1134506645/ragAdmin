# ragAdmin 前端初始化设计

## 1. 目标

本次前端初始化采用一个最小但完整的垂直切片，交付范围限定为：

- 登录页
- 管理台基础框架
- 路由守卫
- 统一 API 封装
- 知识库列表页

本次不包含：

- 知识库创建/编辑
- 文档上传
- 模型管理页面
- 动态菜单与按钮级权限
- 复杂主题切换与国际化

目标是尽快形成可联调、可继续迭代的 `rag-admin-web` 工程基线。

## 2. 方案选择

### 方案一：只做前端空壳

仅初始化登录页、Layout、路由与请求封装。

优点：

- 交付最快

缺点：

- 没有真实业务链路
- 很多联调问题会延后暴露

### 方案二：初始化并带一个真实业务页

在前端基础能力之上，追加知识库列表页。

优点：

- 能跑通登录、认证、表格分页、接口错误处理、菜单高亮整条链路
- 仍然保持较小改动面

缺点：

- 比空壳方案多一页列表页开发量

### 方案三：初始化并直接带列表加创建

优点：

- 更接近演示版本

缺点：

- 初始化阶段容易扩大范围
- 会把更多表单、校验、模型下拉联调问题提前堆叠

最终采用方案二。

## 3. 工程结构

前端新建目录 `rag-admin-web`，采用：

- Vite
- Vue 3
- TypeScript
- Vue Router
- Pinia
- Element Plus
- Axios

建议目录：

- `src/api`：接口封装
- `src/router`：路由定义与守卫
- `src/stores`：认证与应用状态
- `src/layouts`：后台布局
- `src/views`：页面
- `src/components`：复用组件
- `src/styles`：全局样式与主题变量
- `src/types`：通用类型定义
- `src/utils`：Token、本地存储、请求工具

## 4. 页面设计

### 4.1 登录页

登录页只支持当前后端设计中的 `loginId + password`。

页面要求：

- 输入账号
- 输入密码
- 登录按钮
- 错误提示
- 加载态

视觉方向：

- 不做通用后台模板风格
- 采用偏“编辑台 / 知识工坊”的浅色设计
- 背景加入柔和网格、纸面纹理和少量橙色强调色

### 4.2 管理台基础框架

采用左侧导航 + 顶部栏结构。

首批只放两个菜单：

- 概览
- 知识库管理

顶部栏包含：

- 当前用户展示
- 退出登录入口

### 4.3 知识库列表页

首版只做：

- 知识库列表展示
- 分页
- 状态标签
- 空态
- 手动刷新

暂不做：

- 搜索筛选
- 创建按钮
- 详情页跳转

## 5. 数据流设计

### 5.1 认证流

1. 用户访问受保护页面时先经过路由守卫
2. 若本地无 `accessToken`，跳转 `/login`
3. 登录成功后保存 `accessToken`、`refreshToken` 与当前用户摘要
4. 后续请求统一带上 `Authorization: Bearer <token>`
5. 若接口返回未授权，尝试调用刷新接口
6. 刷新成功则重放请求
7. 刷新失败则清空本地状态并跳回登录页

### 5.2 接口响应处理

统一适配后端结构：

- `code`
- `message`
- `data`

页面层只处理解包后的 `data`，不直接解析原始 `AxiosResponse`。

### 5.3 知识库列表页数据来源

使用 `GET /api/admin/knowledge-bases`。

首版参数：

- `pageNo`
- `pageSize`

首版不传筛选条件，保持最小联调路径。

## 6. 错误处理

- 登录失败时展示后端 `message`
- 刷新失败时统一回退到登录页
- 页面初始化失败时提供 `ElMessage` 提示
- 列表接口异常时展示错误占位和重试入口

## 7. 验证范围

本次至少完成以下验证：

- 前端工程可以启动
- 前端工程可以构建
- 登录页路由可访问
- 未登录访问受保护页面会跳转登录页
- 登录成功后可以进入知识库列表页
- 知识库列表页能正确展示后端分页数据

## 8. 后续迭代建议

完成本次基线后，建议按以下顺序继续：

1. 知识库创建页
2. 文档上传与列表
3. 模型管理
4. 系统健康检查页
