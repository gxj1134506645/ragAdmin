# 访问边界收口计划

## 1. 计划定位

本文档统一收口 `ragAdmin` 当前与访问边界相关的方案与实施约束，覆盖以下主题：

- 后台权限边界与在线会话治理
- 统一用户源与前后台登录边界
- 角色、登录入口、登录态、在线会话与强制下线的整体约束

本文档是当前访问边界主题的唯一主 plan，替代此前分散的设计稿与实施计划。

## 2. 背景

当前项目已经明确以下事实：

- 前后台共享统一用户源，不拆分两套账号体系
- 认证与授权统一基于 `sa-token + Redis`
- 登录态、在线会话热数据、强制下线能力由 `sa-token + Redis` 承担
- 聊天历史、知识库内会话与消息事实数据持久化到 PostgreSQL

当前需要进一步收口的重点是：

- 前后台登录入口边界
- 后台权限码矩阵
- 后台在线会话治理
- 用户可进入系统边界的角色控制

## 3. 当前有效结论

### 3.1 统一用户源

- 不新增独立前台账号体系
- 继续复用 `sys_user / sys_role / sys_user_role`
- 后台管理台与问答前台共享统一用户源

### 3.2 登录入口边界

- 后台入口与前台入口分开
- 两个入口都基于统一用户源做账号密码校验
- 能否进入某个入口由角色决定，而不是由是否存在单独账号体系决定

### 3.3 认证与登录态

- 后端认证与授权统一使用 `sa-token`
- Redis 承担登录态、在线会话热数据和强制下线能力
- 前后台登录态按 `loginType` 隔离
- 后台与前台会话热数据不能混用

### 3.4 在线会话治理

- 后台需要具备在线会话查询能力
- 后台需要支持按 `userId` 或会话维度强制下线
- 登录热数据与聊天历史持久化数据职责分离

### 3.5 权限控制

- 后台权限采用显式权限码矩阵
- 后端接口鉴权统一优先采用 `sa-token` 注解与统一权限服务
- 前端统一通过 `/api/admin/auth/me` 获取 `roles + permissions`
- 菜单、路由和高风险按钮统一按权限码收口

## 4. 实施范围

### 4.1 后端

- 接入 `sa-token` 基础设施与 Redis 适配
- 明确 `admin` 与 `app` 两类登录入口
- 收口统一账号密码校验逻辑
- 落地角色与登录入口的放行判断
- 返回后台权限码矩阵
- 补齐在线会话查询与强制下线能力
- 补齐 `APP_USER` 等基础角色数据

### 4.2 前端

- 后台前端按 `permissions` 收口菜单、路由和高风险按钮
- 后台补齐用户管理页面
- 前台与后台分别走各自登录入口，不再共享单一入口语义

## 5. 验收标准

- 前后台使用不同登录入口
- 同一用户源下，角色决定是否允许进入后台或前台
- 后台接口能基于权限码稳定鉴权
- 后台能查看在线会话并执行强制下线
- 登录态隔离稳定，不出现前后台串登录态

## 6. 删除与回写规则

- 本主题后续新增计划，优先追加或改写本文档，不再拆成并列小 plan
- 一旦访问边界结论升级为长期稳定约束，应及时回写到 `docs/architectures`
