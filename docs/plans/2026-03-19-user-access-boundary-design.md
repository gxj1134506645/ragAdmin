# 统一用户源与前后台登录边界设计

## 1. 背景

当前 `ragAdmin` 已经具备统一用户表、角色表、JWT 与 Redis 会话能力，问答前台 `rag-chat-web` 也已经接入 `/api/app/auth/login`。

但当前实现还存在三个明显缺口：

1. `/api/admin/auth/login` 与 `/api/app/auth/login` 复用同一套登录逻辑，没有按角色区分登录入口
2. 后台 Web 还没有用户管理页面，无法直接维护账号、密码、状态与角色
3. 设计文档中已经定义了 `APP_USER`，但 Flyway 初始化脚本尚未补齐这个角色基础数据

这会导致前后台登录边界不清晰，无法形成“统一用户源、按角色放行不同入口”的产品形态。

## 2. 目标

本轮目标是建立最小且可落地的统一账号方案：

- 不新增独立前台账号体系
- 继续复用 `sys_user / sys_role / sys_user_role / JWT / Redis`
- 聊天前台与后台管理台使用不同登录入口，但都基于统一用户源
- 使用角色控制用户可进入的系统边界
- 后台补齐用户管理页面，支持维护账号、密码、状态与角色

## 3. 可选方案

### 方案 A：前后台彻底拆成两套用户体系

优点：

- 逻辑隔离最彻底
- 前后台权限模型不会相互影响

缺点：

- 账号、密码、状态、会话、审计都会重复建设
- 与当前“统一用户源”的既定方向冲突
- 对内部组织使用场景明显过重

### 方案 B：统一用户源，按角色控制前后台登录入口

优点：

- 与当前架构和文档方向一致
- 不需要重建鉴权基础设施
- 用户只维护一份账号资料，易于治理
- 能自然支持“前台用户”和“后台治理用户”的重叠身份

缺点：

- 需要显式定义前后台的角色放行规则
- 登录边界必须在后端收紧，不能只依赖前端路由

### 方案 C：统一用户源，按权限码而不是角色控制登录入口

优点：

- 后续可以做更细粒度授权

缺点：

- 当前项目还没有完整权限码体系与菜单绑定能力
- 对本轮来说属于明显过度设计

## 4. 结论

采用 **方案 B：统一用户源，按角色控制前后台登录入口**。

原因：

- 与当前架构文档、问答前台设计和统一用户源要求一致
- 可以最小成本把“聊天白名单”和“后台治理权限”同时落地
- 不会引入第二套账号体系，后续公司二开也更容易扩展

## 5. 角色与登录边界

本轮定义以下角色语义：

- `APP_USER`：允许登录问答前台
- `ADMIN`：允许登录后台管理台，同时默认允许登录问答前台
- `KB_ADMIN`：允许登录后台管理台，同时默认允许登录问答前台
- `AUDITOR`：允许登录后台管理台，同时默认允许登录问答前台

对应登录规则：

- `/api/app/auth/login`
  - 允许 `APP_USER`
  - 也允许 `ADMIN`、`KB_ADMIN`、`AUDITOR`
- `/api/admin/auth/login`
  - 仅允许 `ADMIN`、`KB_ADMIN`、`AUDITOR`

这样可以满足两个目标：

1. 普通组织成员只授予 `APP_USER` 即可进入聊天前台
2. 后台治理人员天然也能使用前台问答，不需要强制额外再绑一次 `APP_USER`

## 6. 后端改造范围

### 6.1 鉴权服务

在 `AuthService` 中补充“按入口校验角色”的登录方法，避免前后台自己散落判断。

原则：

- 账号、密码校验仍保持统一
- 登录入口的角色放行逻辑集中在认证域
- 未命中允许角色时，返回明确的未授权错误信息

### 6.2 前台门户服务

`AppPortalService.login` 不再直接走“无边界统一登录”，改为调用“前台登录”专用方法。

### 6.3 后台登录入口

`AuthController.login` 改为调用“后台登录”专用方法，确保只有管理角色能进入后台。

### 6.4 Flyway 迁移

新增迁移脚本，补齐 `APP_USER` 基础角色数据，使用非破坏性 `INSERT ... ON CONFLICT DO NOTHING`。

## 7. 后台 Web 改造范围

后台新增“用户管理”页面，支持：

- 用户列表
- 按关键词/状态查询
- 新增用户
- 编辑用户基础资料
- 重置或更新密码
- 配置角色

首轮不做的内容：

- 菜单级权限控制
- 数据权限
- 组织部门树
- 注册审批与邀请流程

## 8. 前端行为预期

### 8.1 问答前台

- 仍使用账号或手机号 + 密码登录
- 如果当前用户没有前台允许角色，登录时直接被后端拒绝

### 8.2 后台管理台

- 仍使用账号或手机号 + 密码登录
- 如果当前用户只有 `APP_USER`，则无法进入后台

## 9. 测试重点

后端重点验证：

- 只有 `APP_USER` 的用户可以登录前台、不能登录后台
- 只有 `ADMIN` 的用户可以登录后台，也可以登录前台
- 状态为 `DISABLED` 的用户不能登录任一入口
- `APP_USER` 角色迁移可重复执行

前端重点验证：

- 后台新增用户后可直接配置 `APP_USER`
- 用户角色变化后，对应登录入口行为符合预期
- 后台用户管理页面的新增、编辑、角色分配链路正常

## 10. 风险与控制

### 风险 1：默认管理员无法进入问答前台

控制：

- 前台登录允许后台治理角色直接进入，因此默认管理员不会被锁死

### 风险 2：只在前端做按钮隐藏，后端接口仍可越权访问

控制：

- 本轮边界收口以服务端登录放行为准，前端展示只是辅助，不作为安全边界

### 风险 3：文档与实际迁移再次不一致

控制：

- 本轮同时补迁移脚本，不再只停留在文档层
