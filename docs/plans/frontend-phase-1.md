# 前端一期阶段计划与执行记录

## 1. 文档目的

本文档用于汇总前端一期阶段的设计决策与执行记录，统一替代初始化设计和当天按小功能拆分产生的多个细粒度 plan 文档。

本阶段文档覆盖的能力范围包括：

- 登录页、管理台基础框架、路由守卫、统一 API 封装
- 知识库创建、编辑、详情
- 知识库文档列表、上传入口、解析触发
- 任务监控、任务重试、任务详情
- 任务与文档的保守联动
- `Vite` 前端拆包优化

后续同类小功能不再单独新增 plan 文档，默认优先追加到阶段文档中。

## 2. 阶段目标

本阶段目标是把管理端前端从“最小可联调基线”推进到“能围绕知识库、文档、任务形成基础操作闭环”。

本阶段完成后，应至少具备以下能力：

- 登录页、后台布局、路由守卫、认证状态管理
- 统一 API 封装与 Token 刷新链路
- 知识库列表页
- 知识库的创建、编辑、详情浏览
- 文档接入知识库并触发解析
- 基础任务可观察与失败任务重试
- 任务、知识库、文档三条线开始建立关联

## 3. 初始化基线补充

### 3.1 初始垂直切片目标

前端初始化采用“最小但完整的垂直切片”，首批先落以下基础能力：

- 登录页
- 管理台基础框架
- 路由守卫
- 统一 API 封装
- 知识库列表页

这样做的目的是先跑通登录、认证、页面访问、列表查询和基础错误处理，不在初始化阶段一次性堆入过多表单和业务写操作。

### 3.2 初始化方案选择

初始化阶段曾评估三种路径：

- 只做前端空壳
- 初始化并带一个真实业务页
- 初始化并直接带列表加创建

最终采用“初始化并带一个真实业务页”，即以知识库列表页作为第一条真实业务链路。原因是它能更早暴露联调问题，同时把改动面控制在较小范围内。

### 3.3 工程结构基线

前端工程目录为 `rag-admin-web`，初始化时采用以下技术组合：

- `Vite`
- `Vue 3`
- `TypeScript`
- `Vue Router`
- `Pinia`
- `Element Plus`
- `Axios`

约定的基础目录包括：

- `src/api`
- `src/router`
- `src/stores`
- `src/layouts`
- `src/views`
- `src/components`
- `src/types`
- `src/utils`

### 3.4 初始化页面与数据流基线

初始化阶段确定了以下基础约定：

- 登录页仅支持后端既定的 `loginId + password`
- 后台采用左侧导航 + 顶部栏结构
- 统一适配后端 `code`、`message`、`data` 响应结构
- 本地无 `accessToken` 时由路由守卫重定向到 `/login`
- 遇到未授权时优先尝试刷新 Token，失败后回退登录页

### 4.1 本阶段包含

- 登录与基础后台布局
- 知识库列表页
- 知识库创建页
- 知识库编辑页
- 知识库详情页
- 知识库文档列表
- 单文件上传接入知识库
- 手动触发文档解析
- 任务监控页
- 失败任务重试
- 任务详情页
- 任务详情中的保守业务关联信息展示

### 4.2 本阶段不包含

- 更激进的前端包体优化

## 5. 关键设计决策

### 5.1 路由结构

当前前端采用独立页面路由，而不是弹窗堆叠式工作流：

- `/knowledge-bases`
- `/knowledge-bases/create`
- `/knowledge-bases/:id`
- `/knowledge-bases/:id/edit`
- `/tasks`
- `/tasks/:id`

该决策的原因是：

- 页面边界清晰
- 后续扩展详情、编辑、版本、切片、任务联动更自然
- 不会让列表页或详情页因为功能累加迅速失控

### 5.2 模型选择策略

知识库创建与编辑中的聊天模型、Embedding 模型采用“双模式兼容”：

- 优先读取真实模型列表
- 接口失败时退化为平台默认模型兜底

设计目标是保证知识库主流程不被模型管理接口拖死。

### 5.3 文档接入策略

文档接入采用三段式闭环：

1. 获取上传凭证
2. 预签名地址直传对象存储
3. 新增文档记录

此处前端不使用统一业务 `http` 客户端上传文件，而是用独立 `fetch` 完成对象存储直传，避免鉴权和统一响应解包干扰预签名上传。

### 5.4 文档解析触发策略

文档解析触发放在知识库详情页文档列表中，以行级按钮发起：

- 仅 `PENDING` / `FAILED` 允许触发
- 只锁当前行，不阻塞整个表格
- 成功或失败后刷新文档列表，以后端真实状态为准

### 5.5 任务监控与治理策略

任务页先做最小可观察性，再逐步引入治理动作：

- 先有列表、筛选、分页、刷新
- 再补失败任务重试
- 再补任务详情

这样可以保证每一轮都有真实用户价值，而不是一次性堆出大而空的“任务中心”。

### 5.6 任务联动策略

任务详情中的业务关联遵循“能确定再展示”的原则：

- 当前能明确识别 `DOCUMENT_PARSE` 类型时，保守展示其关联文档 ID
- 不在前端强行推断知识库或其他对象

该决策是为了避免后端字段尚未稳定时出现错误跳转。

### 5.7 包体优化策略

`Vite` 构建采取“最小风险拆包”而不是激进重构：

- `framework`：`vue`、`vue-router`、`pinia`
- `ui`：`element-plus`、`@element-plus/icons-vue`
- `http`：`axios`

同时将 `chunkSizeWarningLimit` 调整为与当前依赖规模匹配的阈值，目标是先消除构建告警并保持拆包稳定。

## 6. 各能力实现要点

### 6.1 知识库

- 创建页与编辑页复用同一表单组件
- 列表页支持创建回跳刷新与编辑回跳刷新
- 详情页作为知识库工作台承载文档列表
- 文档列表筛选条属于知识库详情页的局部状态，不影响主体信息区

### 6.2 文档

- 当前只做文档接入与解析触发闭环
- 文档详情页采用独立路由 `/documents/:id`
- 文档详情页首版展示主体信息与版本列表，切片浏览后置
- 文档类型通过文件扩展名做前端保守映射

### 6.3 任务

- 任务监控页提供基础筛选与状态颜色区分
- 失败任务支持重试确认与行级 loading
- 任务详情页完整展示错误信息和基础字段
- 任务详情页状态流转时间线采用固定轨迹展示，不伪造中间节点时间

## 7. 验证基线

本阶段代码变更默认至少满足：

- 页面路由可访问
- 关键交互链可跑通
- `npm run build` 通过
- 若涉及构建优化，应明确检查告警是否消失或收敛

## 8. 后续优先级建议

在本阶段基础上，后续前端优先建议如下：

1. 文档版本列表与激活操作增强
2. 文档切片浏览
3. 任务状态流转时间线
4. 任务与文档 / 知识库详情强联动

## 9. 文档收敛规则

从本文档开始，前端一期内的常规小功能不再单独创建 plan 文档。

默认做法：

- 当前阶段内的小功能设计，追加到本文档对应章节
- 只有跨模块、高风险、长周期专题，才允许单独开新文档
- 若某阶段文档过长，再按“前端二期”“后端一期”这类阶段重新拆分

## 10. 文档详情页补充记录

### 10.1 范围

本轮补齐文档详情基础可读页，覆盖：

- 独立文档详情路由 `/documents/:id`
- 文档主体信息展示
- 文档版本列表展示
- 从知识库详情页文档列表进入文档详情页
- 从任务详情页在可识别文档解析任务时进入关联文档

### 10.2 设计要点

- 文档主体详情失败时使用页面级错误态
- 版本列表失败时使用区块级错误态，不影响主体信息展示
- 文档版本激活采用“确认 + 行级 loading + 刷新详情与版本列表”的写操作闭环
- 任务详情到文档详情的跳转仍遵循“能确定再跳”的原则

## 11. 文档版本激活补充记录

### 11.1 范围

本轮补齐文档版本激活最小写操作，覆盖：

- 版本列表非当前版本的“设为生效版本”按钮
- 点击前确认
- 当前行 loading
- 成功后刷新文档详情与版本列表

### 11.2 设计要点

- 当前生效版本不再显示激活按钮
- 前端不本地手改激活状态，而是以后端刷新结果为准
- 激活失败直接展示后端返回文案

## 12. 文档新增版本补充记录

### 12.1 范围

本轮补齐文档新增版本最小闭环，覆盖：

- 文档详情页“新增版本”按钮
- 新增版本弹窗
- 获取上传凭证
- 对象存储直传
- 调用新增版本接口
- 成功后刷新文档详情与版本列表

### 12.2 设计要点

- 新增版本采用弹窗交互，不把上传表单直接堆在版本列表区
- 上传链路沿用“凭证 -> 直传 -> 业务登记”的三段式
- 当前不自动激活新版本
- 当前不引入版本差异比较与变更说明字段

## 13. 知识库文档筛选补充记录

### 13.1 范围

本轮补齐知识库详情页文档区的基础筛选，覆盖：

- `keyword`
- `parseStatus`
- `enabled`
- 查询与重置
- 与分页联动

### 13.2 设计要点

- 筛选条只作用于文档列表，不进入知识库主体区
- 点击查询时回到第一页
- 点击刷新时保留当前筛选与当前分页
- 文档区失败不影响知识库主体区

## 14. 任务状态流转时间线补充记录

### 14.1 范围

本轮补齐任务详情页的状态流转可视化，覆盖：

- 固定状态轨迹展示
- 当前状态高亮
- 已到达 / 未到达区分
- 无精确时间时的占位文案

### 14.2 设计要点

- 时间线固定为 `WAITING -> RUNNING -> SUCCESS / FAILED / CANCELED`
- 前端只基于当前 `taskStatus`、`createdAt`、`updatedAt` 做保守推导
- 无法确定的中间节点时间统一显示“时间未知”
- 不引入步骤树或动态日志面板

## 15. 模型管理页补充记录

### 15.1 范围

本轮补齐模型管理最小可交付页，覆盖：

- 独立模型管理路由 `/models`
- 模型提供方列表
- 模型列表与基础筛选
- 提供方探活与模型探活
- 新增提供方弹窗
- 新增模型弹窗

### 15.2 设计要点

- 页面采用“单页双区块”而不是拆成两个独立管理页
- 上半区管理提供方，下半区管理模型定义，降低导航和联调成本
- 当前只做列表、筛选、探活和新增，不引入编辑、删除和批量操作
- 知识库页面仍兼容原有 `listModels` 调用口径，避免为了模型管理页扩散不必要重构

### 15.3 探活结果增强补充

本轮对模型管理页继续做了最小增强：

- 提供方探活结果补充命中模型编码
- 模型探活与提供方探活结果都绑定到最近一次操作对象
- 探活结果从简单标签改为带状态和消息的明细卡片

设计目标是提升联调和排障时的可诊断性，同时不引入额外子页面或复杂折叠交互。

## 13. 模型管理补充记录

### 13.1 范围

本轮在既有模型管理页上继续补齐最小闭环，覆盖：

- 模型定义编辑
- 模型定义删除
- 前端表单对模型类型与能力类型的强约束
- 后端对模型类型与能力类型关系的服务端校验

本轮不新增独立文档，继续收敛到本阶段文档。

### 13.2 收口策略

- 模型管理页继续保留为统一入口，不再拆出单独“修改模型配置”模块
- 编辑与新增复用同一弹窗，减少重复交互和状态分叉
- 删除模型前先做引用检查，避免破坏知识库、向量索引和历史对话数据
- 对 `CHAT` / `EMBEDDING` 模型分别锁定唯一合法能力，避免把聊天模型误配成向量模型

### 13.3 结果

- 前端已支持在模型表格中直接编辑、删除、探活
- 后端已提供 `PUT /api/admin/models/{id}` 与 `DELETE /api/admin/models/{id}`
- 如果模型仍被知识库、向量引用或历史对话使用，删除会被显式拒绝
- `deepseek-v3.2` 这类聊天模型后续在表单中默认只能保留“文本生成”能力

## 16. 用户管理页信息密度收敛补充记录

### 16.1 范围

本轮对用户管理页做信息密度收敛，重点把分页列表前置，覆盖：

- 收缩顶部统计卡数量与高度
- 移除独立角色说明大卡
- 将筛选区并入用户列表主面板
- 强化“分页结果总数 / 当前页范围”提示

### 16.2 设计要点

- 页面结构从“页头 + 多层说明卡 + 列表”收敛为“页头 + 紧凑摘要 + 列表主区”
- 角色说明不再占据独立整排卡片，改为列表头部的紧凑标签提示
- 分页相关信息优先展示检索结果总数、当前页范围和页内启用数量，避免用户视线先落在次级说明区
- 筛选栏保留原有查询能力，但直接挂在表格上方，减少滚动距离并强化“筛选即作用于下方列表”的感知

### 16.3 二次收口调整

根据页面实看反馈，本轮继续做了更激进的收口：

- 删除顶部分页概览与在线统计卡，避免列表再次被说明信息挤压
- 删除“列表重点”“用户分页列表”等大标题，只保留一行结果提示
- 将“刷新列表”“新增用户”下移到查询条件区下方，与筛选动作合并成统一操作带
- 用户表格新增“在线状态”列，直接展示“前台在线 / 后台在线 / 双端在线 / 全部离线”

本轮的目标是让用户进入页面后第一视觉焦点直接落在分页结果本身，而不是落在说明卡和统计卡上。

## 17. 后台问答入口下线补充记录

### 17.1 范围

本轮按产品边界收口后台管理端，覆盖：

- 删除后台左侧“智能问答”菜单
- 下线后台独立 `/chat` 问答页
- 删除首页快捷入口中的后台问答卡片
- 移除知识库详情页内嵌问答工作台

### 17.2 设计要点

- 后台管理端聚焦治理，不再承载独立 AI 问答主场景
- 问答能力统一回收到 `rag-chat-web` 前台入口，减少管理端与使用端职责混杂
- 后台旧 `/chat` 路径保留重定向兼容，避免历史书签直接进入 404
- 知识库详情页保留知识库配置、文档治理和任务联动，不再内嵌对话工作台
