# ragAdmin 开发规范

## 第一性原则

这是本仓库的最高优先级沟通与输出规则：
- 任何时刻，必须使用中文简体与用户沟通
- 任何 Markdown 内容输出，必须默认使用中文简体
- 除非用户明确要求英文，否则不得切换为英文回复
- 技术名词、接口路径、类名、表名、命令、配置键名可保留原始英文，但解释、说明、步骤、结论必须使用中文简体

这条规则优先于一般文档风格偏好，后续所有补充规范都不得与其冲突。

## 项目概述

`ragAdmin` 是一个面向内部使用的 RAG 知识库管理平台。

当前仓库状态：
- 还没有正式业务源码
- 现阶段以架构设计和项目启动为主
- 当前设计文档位于 `docs/`

首期目标：
- 知识库与文档管理
- 文档异步解析与索引构建
- 模型与提供方管理
- 单知识库 RAG 问答
- 后台治理能力：认证、权限、审计、任务监控

## 推荐技术栈

除非用户明确指定，否则优先采用以下组合：
- 后端：JDK 21 + Spring Boot 3
- 前端：TypeScript + Vue 3
- 数据库：PostgreSQL 16
- 向量检索：首期优先 `pgvector`，后续再评估 `Milvus`
- 对象存储：MinIO
- AI 接入：Spring AI Alibaba，但外层必须封装自定义模型适配层
- 数据库迁移：Flyway
- 定时任务：如果当前形态是单体项目，优先使用 Spring Boot 自带定时任务；只有在需要持久化、失败补偿、集群调度、错过补触发、任务管理后台时，再升级到 Quartz 或 XXL-JOB

补充默认选型：
- ORM：MyBatis Plus
- 对象映射：MapStruct
- 认证/会话：JWT 优先，并首批接入 Redis 进行会话控制与扩展支撑
- Redis：属于首批基础设施，首期即接入；用于会话控制、热点缓存、限流、防重复提交、任务去重、分布式锁，以及后续多实例部署扩展
- 日志：首批即接入 `SLF4J + Logback`
- 可观测性：首批即具备基础可观测能力，至少包含结构化日志、健康检查、关键任务与失败路径可追踪
- 前端 UI：Element Plus

## 架构约束

必须遵守以下边界：
- 业务层不得直接依赖具体厂商模型 SDK
- Spring AI Alibaba 只能位于适配层/基础设施层，不能渗透到核心领域层
- RAG 检索链路应集中在编排层，不能散落到 Controller 中
- 文档处理必须异步化
- Embedding 模型与向量存储必须可替换

建议逻辑分层：
1. 接入层
2. 领域层
3. 编排层
4. 基础设施层

核心平台能力：
- Model Gateway
- Knowledge Pipeline
- Retrieval Orchestrator
- Admin Governance

## 开发原则

默认遵循以下原则：
- KISS：优先最简单、能落地的方案
- YAGNI：不为“未来可能会用到”提前做抽象
- DRY：明显重复的业务逻辑要及时收敛
- SOLID：核心模块严格遵守，简单 CRUD 不做形式主义

本项目的优先级顺序：
1. 能交付端到端可运行价值
2. 边界清晰、便于维护
3. 性能与扩展性
4. 框架层面的优雅

早期实现时，优先做“小而完整的垂直切片”，不要先铺一层大而空的脚手架。

## API 规范

当前 API 基线：
- 管理端前缀：`/api/admin`
- 内部回调前缀：`/api/internal`
- 统一响应结构：`code`、`message`、`data`
- 分页字段：`pageNo`、`pageSize`、`total`、`list`
- 认证头：`Authorization: Bearer <token>`

当前状态枚举基线：
- 通用状态：`ENABLED` / `DISABLED`
- 文档解析状态：`PENDING` / `PROCESSING` / `SUCCESS` / `FAILED`
- 任务状态：`WAITING` / `RUNNING` / `SUCCESS` / `FAILED` / `CANCELED`

接口边界规则：
- Controller 入参必须使用请求 VO/DTO
- Controller 出参必须使用响应 VO/DTO
- 禁止直接暴露 Entity 作为 API 契约
- 除非用户明确接受动态结构，否则禁止将 `Map`、`JSONObject`、`Object` 作为公共 API 契约
- 对外接口结构必须显式、可读、可校验

## 数据库规范

当前数据库草案位于 `docs/rag-admin-schema-v1.sql`。

当前默认假设：
- PostgreSQL 16
- 已启用 `pgvector`
- 初始向量维度为 `1024`

如果实现阶段修改了向量维度或存储结构，必须同步更新 SQL 草案和相关设计文档。

数据演进规则：
- 所有表结构变更必须通过 Flyway 管理
- 禁止直接手改生产数据库结构
- 优先使用非破坏性演进，避免直接 destructive change
- 每个 migration 必须具备清晰版本号和变更目的

数据库设计约定：
- 默认使用 `BIGINT` 主键，除非有明确反例理由
- 核心表的审计字段保持一致
- 状态字段和软删除语义必须统一、清晰
- 对高频过滤条件和关联字段有意识地补索引

## Codex 工作规则

新增代码时：
- 以小步快跑方式推进，不做大面积空壳式脚手架
- 目录结构和模块边界要与现有架构文档一致
- 优先保证边界清晰，不追求框架堆砌
- 只引入当前任务真正需要的最小依赖
- 命名尽量与文档术语保持一致：知识库、文档、分段、提供方、模型、路由、审计、任务
- MyBatis Plus 单表查询优先使用 Lambda Wrapper 风格
- Entity 与 VO/DTO 的映射优先使用 MapStruct，不写大段手动拷贝
- 安全、认证、数据边界相关逻辑必须显式可见，不要靠隐式约定
- 在不造成明显噪音的前提下，必须尽可能为代码补充单行或多行注释，注释默认使用中文简体，优先说明边界条件、意图和非直观实现
- 对于 Entity、VO、DTO、Query、Request、Response 这类主要数据对象，默认优先使用 Lombok 注解生成 `getter` / `setter` / `constructor` / `builder` / `chain set`，避免手写重复样板代码
- 如果引入 Lombok，必须同时保证对应依赖和注解处理器配置完整，并注意与 MapStruct 的处理顺序，避免生成代码冲突
- 使用 MapStruct 时，Mapper 接口必须显式配置为可注入 Spring Bean，例如 `componentModel = "spring"`，禁止只保留普通接口而不纳入 Spring 容器
- Bean 注入默认优先使用 `@Autowired` 或 `@Resource`，并在实现时显式检查依赖方向，必须小心避免循环依赖；一旦发现潜在循环依赖，优先调整设计而不是硬性规避

AI 生成代码时必须检查：
- 是否存在注入、反序列化、敏感信息泄露等安全问题
- API 契约是否清晰且类型明确
- 异常处理与日志是否合理
- 是否出现不必要的抽象和过度设计
- 是否在不同模块重复扩散相同逻辑

修改文档或 schema 时：
- 保持术语一致
- 保持架构文档、API 文档、SQL 文档之间的命名一致
- 发现冲突时直接指出，不要静默忽略

代码风格默认值：
- 统一使用 UTF-8
- 需要写注释时，优先使用中文简体
- 默认倾向补充高价值中文简体注释，尤其是复杂分支、状态流转、外部依赖调用和边界处理
- 主要数据对象默认不手写低价值 `getter` / `setter` / `constructor`，优先收敛为 Lombok 注解表达
- 复杂分支、边界条件、非直观设计要补注释
- 不要写只是重复代码字面意思的低价值注释

可观测性默认要求：
- 首批必须具备基础可观测性，不能等到后期再补
- 统一使用 `SLF4J + Logback`
- 对关键操作和失败路径增加结构化日志
- 禁止记录密码、Token、密钥和原始敏感内容
- 对长耗时的解析、索引、任务流转保留可追踪性
- 至少提供健康检查能力，并能快速定位数据库、Redis、MinIO、模型调用等依赖状态

## Markdown 与文档规则

文档编写默认规则：
- 任何 `*.md` 文档默认使用中文简体输出
- `*.md` 文档内容尽量使用中文简体
- 除非有明确必要，不要中英混杂大段描述
- 标题、说明、备注、操作步骤优先使用中文简体
- 面向项目成员的规范、设计、说明文档默认用中文简体
- 对外协议名、技术名词、接口路径、类名、表名可保留英文原文

`AGENTS.md` 本身也应尽量保持中文简体表达，后续追加内容时遵守相同规则。

## 近期实现顺序

推荐实现顺序：
1. 后端项目初始化
2. 认证与基础用户/角色模型
3. 模型提供方与模型定义管理
4. 知识库与文档管理
5. 文档解析任务流
6. 基础 RAG 检索与回答链路
7. 审计与运维支持

## 交付流程

默认执行流程：
1. 明确目标与约束
2. 做最小且有效的改动
3. 在当前环境允许范围内完成验证
4. 总结关键决策和剩余风险

如果工作按阶段推进，每个阶段结束时都应处于“可编译、可评审、可继续迭代”的状态。

每完成一个相对独立的大任务后，默认执行以下 Git 收口动作：
- 先自检并确认工作区只包含本任务相关改动
- 使用符合 Conventional Commits 的中文提交信息完成一次提交
- 如无特殊说明，提交完成后继续推送到远端，保持仓库状态与阶段结果同步

Git 规范：
- 使用 Conventional Commits
- 本仓库提交信息尽量使用中文简体
- 每次提交聚焦一个清晰改动
- 不把无关重构和功能改动强行混在一起

建议使用的提交类型：
- `feat`
- `fix`
- `docs`
- `refactor`
- `test`
- `chore`
- `perf`

AI 协作规则：
- AI 适合生成样板代码、重复映射、测试草稿、初始化结构
- 架构、安全、数据边界、关键设计决策必须人工把关
- 分析要尽量短，结论要尽量可执行
- 以代码和提交历史为主工作载体，只有在确有长期价值时才补额外文档
