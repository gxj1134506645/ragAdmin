# 文档解析触发设计说明

## 1. 目标

本次实现目标是在现有知识库详情页与文档上传能力基础上，补齐“手动触发文档解析”最小闭环：

- 在知识库详情页文档列表中提供“开始解析”入口
- 调用文档解析触发接口
- 对当前文档提供行级 loading 反馈
- 触发成功或幂等提示后刷新文档列表状态

本轮重点是把“上传文档 -> 触发解析 -> 查看状态变化”这条链跑通，而不引入任务中心联动。

## 2. 方案选择

本次采用“知识库详情页文档列表内触发解析 + 行级反馈”方案。

选择原因如下：

- 路径最短，用户在当前知识库上下文中就能完成解析触发
- 行级 loading 更能明确是哪个文档正在触发操作
- 不需要为了当前能力单独再引入抽屉或任务页结构

## 3. 交互设计

在知识库详情页文档列表的“操作”列新增 `开始解析` 按钮。

按钮显示规则：

- `PENDING`：允许触发
- `FAILED`：允许重新触发
- `PROCESSING`：不允许重复触发
- `SUCCESS`：本轮默认不重复触发

交互流程如下：

1. 用户点击某一行文档的 `开始解析`
2. 当前行按钮进入 loading
3. 调用 `POST /api/admin/documents/{documentId}/parse`
4. 成功后提示“已提交解析任务”
5. 随后刷新当前知识库文档列表，以后端真实状态为准

## 4. 数据流设计

解析触发不需要复杂请求体，前端只需按文档 ID 调用接口。

状态更新策略：

- 不在前端本地直接改写 `parseStatus`
- 始终以刷新后的列表数据为准

这样可以避免前端对状态流转做错误假设。

## 5. 错误处理

错误处理分为两类：

- 普通失败：显示明确错误信息
- 幂等类提示：例如已有进行中的解析任务，也提示用户并刷新列表

同时要求：

- 只锁当前行按钮，不阻塞整个表格
- 失败后当前行可再次点击
- 列表刷新失败时仍保留本次操作提示

## 6. 本轮边界

本轮包含：

- 文档列表“开始解析”按钮
- 行级 loading
- 解析触发接口封装
- 成功 / 幂等 / 失败消息提示
- 触发后刷新当前文档列表

本轮不包含：

- 任务列表联动
- 任务详情页入口
- 自动轮询解析状态
- 批量触发解析
- 文档版本粒度解析

## 7. 验证范围

本轮至少完成以下验证：

- 可对符合条件的文档点击“开始解析”
- 成功后可提示并刷新文档列表
- 已处理完成或处理中状态不会重复触发
- 失败或幂等提示场景有明确反馈
- 前端项目可通过 `npm run build`
