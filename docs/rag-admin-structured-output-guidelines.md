# ragAdmin 结构化输出适用场景设计说明

## 1. 文档定位

本文档用于回答 `ragAdmin` 在当前阶段应如何使用结构化输出，并明确以下问题：

- 什么是结构化输出，它与流式输出是什么关系
- 哪些场景适合使用结构化输出，哪些场景不适合
- 在 `ragAdmin` 当前架构中，结构化输出应该落在哪些链路
- 结构化输出的接口形态、提示词约束、异常回退应如何设计
- 如何避免把“前端事件流格式”与“模型结构化结果”混为一谈

当前阶段如果历史讨论、临时实现与本文档冲突，以本文档为准；如后续业务方向变化，应优先更新本文档，再调整实现。

## 2. 当前结论摘要

先给出直接结论：

- 结构化输出不是越多越好，只适合“结果要被程序继续消费”的场景
- 普通自然语言回答本身不适合强制结构化输出，尤其不适合逐 token 流式结构化
- 流式输出解决的是“边生成边返回”，结构化输出解决的是“结果字段可校验、可编排、可机读”，二者不是同一个维度
- `Flux<T>` 或 `Flux<String>` 讨论的是接口传输形态，不等于模型是否结构化输出
- `ragAdmin` 当前最适合结构化输出的场景是：摘要生成、意图识别、检索规划、工具参数生成、回答后处理元数据
- `ragAdmin` 当前不适合结构化输出的场景是：面向用户展示的正文回答全文、逐 token 聊天输出、长篇自由创作类回答

## 3. 基本概念澄清

### 3.1 什么是结构化输出

结构化输出指模型返回的数据被约束为一个明确 schema，而不是单纯一段自由文本。

典型形式包括：

- `SummaryResult { summary: string }`
- `IntentDecision { intent: string, needRetrieval: boolean, needWebSearch: boolean }`
- `RetrievalPlan { rewrittenQuery: string, topK: int, filters: [...] }`
- `AnswerPostProcess { answer: string, confidence: string, citations: [...] }`

结构化输出的重点不是“像 JSON”，而是：

- 字段稳定
- 类型明确
- 程序可校验
- 便于后续编排

### 3.2 什么是流式输出

流式输出指服务端将模型生成结果分片返回，而不是等完整结果生成后一次性返回。

例如：

- 聊天回答 token 流
- 任务状态事件流
- 长耗时过程中的阶段通知流

流式输出的重点是：

- 降低首字节等待时间
- 改善交互体验
- 支持长耗时过程实时反馈

### 3.3 两者的关系

结构化输出和流式输出互相独立：

- 一个接口可以是非流式但结构化，例如“摘要生成接口返回一个对象”
- 一个接口可以是流式但非结构化，例如“聊天正文按文本 delta 推送”
- 一个接口也可以是“前半段流式文本，结尾再补一个结构化事件”，属于混合模式

因此不能把以下两件事混为一谈：

- “后端是否使用 `Flux` / SSE”
- “模型返回是否是结构化 schema”

## 4. 判断一个场景是否适合结构化输出

### 4.1 四个核心判断维度

一个场景越满足以下条件，越适合结构化输出：

1. 结果是否要被程序继续消费，而不是只给人看
2. 字段是否相对稳定，短期内不会高频变化
3. 字段缺失、歧义或格式漂移是否会直接影响后续业务逻辑
4. 能否接受一次性完整返回，而不是必须逐 token 展示

如果以上四项多数为“是”，通常适合结构化输出。

### 4.2 三个典型反向信号

出现以下信号时，通常不适合结构化输出：

- 结果本质上是开放式自然语言正文
- 用户希望边生成边看内容
- 字段设计本身还没有稳定，今天一个 schema、明天又变

### 4.3 简化判断公式

可以用一句话做快速判断：

“这个结果如果不给人看，而是交给下一段程序逻辑继续处理，它还能成立吗？”

如果答案是“能”，结构化输出的价值通常较高。

## 5. 适合做结构化输出的场景

### 5.1 会话摘要生成

适合程度：高

原因：

- 摘要不是直接给用户逐 token 阅读的正文
- 摘要字段天然稳定
- 后续可能要存库、检索、压缩会话、做审计或做二次计算
- 出现字段漂移会直接影响下游编排

当前状态：

- 已落地为单字段结构化输出
- 当前载荷对象为 `ConversationSummaryStructuredOutput`

后续可扩展字段：

- `summary`
- `facts`
- `constraints`
- `todos`
- `risks`

但当前阶段不建议一开始就做复杂多字段，先保留单字段，避免 schema 过早膨胀。

### 5.2 意图识别

适合程度：高

典型用途：

- 判断问题属于知识库问答、通用问答、联网增强问答，还是闲聊
- 判断是否需要检索
- 判断是否需要联网
- 判断是否需要澄清问题

适合原因：

- 这是编排前置决策，不是直接展示给用户的正文
- 结果天然适合布尔字段、枚举字段
- 后续逻辑通常是 `if / switch`，不适合靠自由文本解析

推荐 schema 示例：

```json
{
  "intent": "KNOWLEDGE_QA",
  "needRetrieval": true,
  "needWebSearch": false,
  "needClarification": false,
  "reason": "问题明显指向已接入的知识库主题"
}
```

说明：

- `reason` 可以保留给日志与排查，不建议直接作为强业务依赖
- 核心依赖字段应是枚举和布尔值

### 5.3 检索规划

适合程度：高

典型用途：

- 改写用户查询
- 生成检索关键词
- 决定 `topK`
- 决定是否启用过滤条件
- 决定检索范围或检索模式

适合原因：

- 检索规划天然是中间态结果
- 后续要交给检索服务继续执行
- 如果仍然返回自由文本，后端还得二次解析，风险高

推荐 schema 示例：

```json
{
  "rewrittenQuery": "企业知识库中关于智能体应用落地的最佳实践",
  "topK": 6,
  "filters": [],
  "retrievalMode": "VECTOR",
  "needRerank": false
}
```

注意：

- `topK`、`retrievalMode` 这类字段应有服务端兜底
- 结构化输出只提供建议，不应允许模型绕过平台安全边界

### 5.4 工具参数生成

适合程度：高

典型用途：

- 生成联网搜索参数
- 生成某个内部工具调用入参
- 生成任务重试或调度参数

适合原因：

- 工具调用本质上就是结构化入参
- 这是 LLM 编排里最经典的结构化输出使用点
- 如果不做结构化，很容易出现字段名漂移、值类型错误、漏字段

设计建议：

- 工具参数必须显式白名单化
- 服务端必须做二次校验
- 不允许模型自由传递任意额外字段直达底层执行器

### 5.5 回答后处理元数据

适合程度：中高

典型用途：

- 给回答补充置信度等级
- 标记是否命中知识库证据
- 生成引用片段 ID 列表
- 标记是否需要人工复核

适合原因：

- 面向程序消费
- 适合在正文回答生成完成后再一次性产出
- 可用于前端展示标签、风控提示、质量评估

推荐做法：

- 不要把“正文回答”和“元数据”强行揉成一个复杂大对象再逐 token 输出
- 更适合采用“两阶段模式”：
  - 第一阶段输出正文
  - 第二阶段输出结构化元数据

## 6. 不适合做结构化输出的场景

### 6.1 面向用户的普通正文回答

不适合程度：高

原因：

- 用户关心的是自然语言结果，不是字段对象
- 如果强行结构化，通常会把回答拆成 `answer`、`evidence`、`nextStep` 等字段，反而限制表达
- 前端还要做额外拼装，收益不高

对于普通问答，推荐直接输出自然语言正文。

### 6.2 逐 token 流式聊天正文

不适合程度：高

原因：

- JSON / schema 在逐 token 流式过程中天然不完整
- 前端需要处理半截字段、半截数组、未闭合对象，复杂度很高
- 一旦模型中途插入解释性文本，整个结构就可能失效

这也是为什么“聊天主回答”不应该用结构化输出强行承载。

推荐做法：

- 正文流式输出仍使用 `delta text`
- 如果确实需要结构化元信息，在流结束时单独补一条结构化事件

### 6.3 长篇创作类、开放式发散回答

不适合程度：高

例如：

- 方案阐述
- 解释说明
- 对比分析
- 用户让模型“详细展开”

原因：

- 这些内容的价值在于表达质量，不在于字段稳定
- 强 schema 会压缩模型表达空间
- 最终往往仍要再把字段拼回自然语言

### 6.4 schema 尚未稳定的探索期场景

不适合程度：中高

如果业务方自己还没想清楚字段定义，例如：

- 到底要不要 `needWebSearch`
- `confidence` 是数值还是枚举
- `intent` 粒度怎么划分

那就不应该过早落地结构化输出。

否则会出现：

- prompt 频繁变
- DTO 频繁改
- 前后端联调频繁返工

建议先在日志或实验链路中验证，等字段稳定再固化 schema。

## 7. 半结构化与混合模式

并不是所有场景都只能二选一。

在 `ragAdmin` 中，最实用的通常是以下三种混合模式。

### 7.1 文本正文 + 结构化尾事件

适合：

- 主回答需要流式展示
- 但结束后仍需要程序消费元数据

示例：

- SSE `delta` 事件持续输出回答正文
- SSE `complete` 事件输出持久化结果
- 可再追加一个 `meta` 事件输出：
  - `confidence`
  - `citations`
  - `needFollowUp`

### 7.2 结构化规划 + 自然语言执行

适合：

- 前一阶段需要模型做决策
- 后一阶段需要模型生成自然语言

典型链路：

1. 先做结构化“意图识别 / 检索规划”
2. 再按规划结果执行检索
3. 最后让模型生成自然语言回答

这是当前 `ragAdmin` 最推荐的结构。

### 7.3 自然语言主结果 + 结构化持久化补写

适合：

- 用户只看自然语言结果
- 但后台希望补充结构化摘要、标签、统计信息

典型链路：

1. 用户先拿到自然语言回答
2. 后台异步生成摘要、标签、风险标记并落库

会话摘要就是该模式的典型例子。

## 8. `ragAdmin` 当前架构中的推荐落点

### 8.1 当前已落地点

当前已落地：

- 会话摘要：适合结构化输出，已接入
- 问答规划：适合结构化输出，已接入最小切片，当前覆盖意图识别、检索开关、联网开关、查询改写
- 回答后处理元数据：适合结构化输出，已接入轻量元数据，当前覆盖 `confidence`、`hasKnowledgeBaseEvidence`、`needFollowUp`

当前对应链路：

- `ConversationChatClient.chatEntity(...)`
- `DefaultConversationSummaryGenerator`
- `DefaultChatExecutionPlanningService`
- `DefaultChatAnswerMetadataGenerationService`

### 8.2 近期推荐优先级

推荐优先级从高到低如下：

1. 意图识别
2. 检索规划
3. 回答后处理元数据
4. 工具参数生成
5. 更丰富的摘要字段

理由：

- 这些场景都属于“模型输出先给程序消费”
- 落地后能直接降低编排层的字符串解析风险
- 不会破坏当前主问答流式体验

### 8.3 当前不建议立即改造的链路

当前不建议立即结构化的链路：

- 主聊天回答全文
- 流式 token 输出
- 任务实时事件主数据

说明：

- 任务实时事件本身已经是后端定义好的 DTO，不需要再让模型结构化输出
- 聊天主回答是面向用户的自然语言，不应硬塞进 schema

## 9. 接口与返回形态建议

### 9.1 非流式结构化输出

适用：

- 摘要
- 意图识别
- 检索规划

推荐接口形态：

- 服务内部：`T chatEntity(..., Class<T> responseType)`
- 控制层：常规 JSON 响应或内部方法返回 DTO

### 9.2 流式自然语言输出

适用：

- 主聊天回答
- 长文本生成

推荐接口形态：

- `Flux<ChatStreamEventResponse>`
- 对纯正文 token 流，可进一步收敛为 `Flux<String>`

这里讨论的是“事件传输格式”，不是“模型结构化输出格式”。

### 9.3 混合事件流输出

适用：

- 主回答需要流式
- 结束后还要补结构化元信息

推荐事件类型：

- `delta`
- `complete`
- `meta`
- `error`

其中 `meta` 可承载结构化对象。

## 10. 实现约束

### 10.1 统一走 `ConversationChatClient`

在 `ragAdmin` 中，所有面向聊天模型的业务编排入口应优先走 `ConversationChatClient`。

原因：

- 统一外观
- 统一日志
- 统一模型工厂
- 统一记忆与无记忆路径划分

不应再回退到分散的厂商适配类直接调用。

### 10.2 有记忆与无记忆路径必须分开

结构化输出并不一定需要会话记忆。

例如：

- 会话摘要
- 意图识别
- 检索规划

这些场景更适合无状态调用。

否则容易出现：

- 历史 memory 污染当前结构化结果
- 摘要被旧上下文意外干扰
- 意图判断受无关历史噪声影响

### 10.3 schema 必须由服务端控制

结构化输出对象必须是服务端显式定义的 DTO / record，而不是把任意 JSON 当成黑盒字符串处理。

禁止做法：

- `Map<String, Object>` 直接承接公共结构化结果
- 前端自己猜字段
- 模型自由生成未声明字段再由下游逻辑直接使用

### 10.4 必须设计回退策略

结构化输出不是 100% 稳定，必须有兜底。

至少要考虑：

- 模型返回为空
- 字段不完整
- 解析失败
- 模型偶尔退化成自然语言文本

推荐回退方式：

- 摘要失败时回退规则摘要
- 意图识别失败时回退保守默认值
- 检索规划失败时回退固定 `topK + 原始 query`

## 11. 提示词与 schema 设计建议

### 11.1 字段越少越稳定

如果一个场景只需要三个字段，就不要设计八个字段。

字段过多会带来：

- prompt 更长
- 解析更脆弱
- 模型遗漏字段概率更高
- 维护成本更高

### 11.2 枚举优先于自由文本

例如优先用：

- `intent = KNOWLEDGE_QA | GENERAL_QA | CLARIFICATION`

而不是：

- `intentDescription = "这个问题看起来像是知识库问答"`

自由文本可以作为辅助说明，但不能作为主决策字段。

### 11.3 解释字段不能成为强依赖

像 `reason`、`analysis`、`explanation` 这类字段适合记录日志、帮助排查，不适合作为核心控制逻辑条件。

业务逻辑应依赖：

- 布尔值
- 枚举
- 数值
- 明确 ID 列表

## 12. 与前端协议的边界

### 12.1 前端不应承担模型结构化纠错职责

如果后端决定使用结构化输出，那么：

- schema 校验
- 解析失败处理
- 回退策略

都应由后端完成。

前端只消费已经稳定的接口契约，不应再去猜测模型原始输出是否合法。

### 12.2 流式复杂度的核心不在 Java 泛型包装

即使控制器直接返回 `Flux<T>`，在 `text/event-stream` 下浏览器拿到的仍然是标准事件流。

真正的复杂度不在于有没有 `ServerSentEvent` 包装，而在于：

- 事件类型是否清晰
- `data` 内部字段是否稳定
- 是否存在多种互相冲突的事件模型

因此当前推荐直接使用 `Flux<T>`，并把业务语义稳定收敛到数据体字段中，例如 `eventType`、`messageId`、`usage`。

## 13. 推荐落地顺序

### 13.1 第一阶段

- 会话摘要结构化输出

状态：

- 已完成

### 13.2 第二阶段

- 意图识别结构化输出

目标：

- 判断是否需要检索
- 判断是否需要联网
- 判断是否需要澄清

### 13.3 第三阶段

- 检索规划结构化输出

目标：

- query rewrite
- topK 建议
- 检索模式建议
- 过滤条件建议

### 13.4 第四阶段

- 回答后处理元数据

目标：

- 置信度标签
- 引用状态
- 风险标记

## 14. 最终原则

最后收敛为五条原则：

- 只在“程序要消费结果”时优先考虑结构化输出
- 不要把流式输出问题和结构化输出问题混为一类
- 主聊天正文默认保持自然语言，不要强制结构化
- 结构化输出对象必须小而稳定，并由服务端显式控制
- 所有结构化输出都必须有回退路径，不能把模型 schema 遵守能力当成绝对前提

以上原则适用于 `ragAdmin` 当前阶段的问答、检索、摘要和模型编排链路。
