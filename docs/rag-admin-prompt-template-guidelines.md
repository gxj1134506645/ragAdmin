# ragAdmin 提示词模板化与现实世界问答优化经验

## 1. 文档定位

本文档用于沉淀 `ragAdmin` 在提示词设计、模板管理和现实世界问答增强上的一组长期有效经验，重点回答以下问题：

- 为什么不应把大段 prompt 继续硬编码在 Java Service 中
- 为什么“优化提示词”不能退化成“遇到一个 case 补一句提示”
- 什么时候应该做输入归一化，什么时候应该引入专用 tool / MCP
- 提示词模板在后端项目中应该如何组织、注入、渲染和测试

当前阶段如果局部实现与本文档冲突，应优先更新本文档，再调整实现。

## 2. 这次问题暴露出的真实症结

本次问题不是“用户问得不清楚”，也不是“模型单纯变蠢了”，而是编排链路缺少足够稳定的现实世界上下文。

典型现象：

- 用户提问“今天无锡天气如何”
- Tavily 已成功返回联网结果
- 模型却仍然追问“今天指的是哪一天”

这说明：

1. 搜索成功不等于回答稳定
2. 模型看到联网摘要后，仍然可能因为时间上下文不完整而做保守退缩
3. 问题不在某一条结果，而在问答编排层没有把“当前日期、时区、相对时间语义”明确传给模型

## 3. 不应继续采用的坏模式

### 3.1 遇到一个 case，补一句提示词

这是最危险的 prompt 演进方式。

坏处包括：

- prompt 越来越长，但设计边界越来越不清晰
- 每次修复只面向单一 case，无法形成可复用能力
- 最终会出现一份“什么都想管”的大 prompt，但没有任何层次结构
- 维护者很难判断某一句提示到底服务于什么场景，删除也不敢删

### 3.2 把现实世界问题一股脑交给通用联网摘要

天气、新闻、股价、汇率、节假日、航班、地图、政策时效信息，本质上都属于强现实世界约束场景。

如果只把“搜索结果摘要”拼给模型，而不提供：

- 当前日期
- 当前时区
- 用户所在地区
- 需要回答的是哪一天、哪一个时间窗口

模型就只能从摘要中猜，猜错并不意外。

## 4. 正确分层：不要把所有问题都归因于 prompt

现实世界问答要稳定，至少需要三层分工。

### 4.1 输入归一化层

这一层负责把用户自然语言中的隐式上下文显式化、结构化。

典型任务：

- 将“今天 / 明天 / 昨天”转换为绝对日期
- 将“本周 / 下周 / 本月底”转换为明确时间区间
- 将“这里 / 当地 / 附近”绑定到显式地区
- 将“最新 / 刚刚 / 当前”标记为强时效查询

这类逻辑不是针对“天气”单独硬编码，而是通用上下文归一化能力。

### 4.2 能力路由层

这一层负责决定问题应该走什么能力，而不是默认都走大模型自由发挥。

建议路由规则：

- 天气、股价、汇率、地图、日历、航班、快递等高时效强约束问题：
  优先使用专用 tool / MCP / 外部 API
- 通用公开资讯、行业动态、政策新闻：
  可优先走联网搜索增强
- 内部制度、流程、知识库事实：
  优先走知识库检索

### 4.3 Prompt 模板层

Prompt 的职责应收敛为：

- 表达回答原则
- 引导模型如何使用已经准备好的上下文
- 控制输出边界与格式

Prompt 不应承担：

- 所有现实世界事实判断
- 所有工具路由逻辑
- 所有隐式时间解释

换句话说：

能前置归一化的，不要全丢给 prompt；能走专用 tool 的，不要硬逼模型从搜索摘要里猜。

## 5. 本次可积累的关键经验

### 5.1 相对时间归一化是通用能力，不是天气特判

“今天无锡天气如何”只是把问题暴露得更明显。

同样的能力还会服务于：

- 今天 AI 行业有什么新动态
- 明天上海有雨吗
- 昨天金价涨了吗
- 本周有哪些政策更新
- 现在这个时间段适合发版吗

因此，相对时间归一化应归类为通用输入标准化能力，而不是某个单点补丁。

### 5.2 搜索 query 也要做时间归一化

如果前端问题里有“今天”，而发给搜索引擎的 query 仍然保留“今天”，就更容易召回多天混合页面。

实践结论：

- 用户问题保留原始语义
- 但发给联网搜索 Provider 的查询词可以做“相对时间 -> 绝对日期”的归一化

例如：

- 用户问题：`今天无锡天气如何`
- 归一化搜索词：`2026-03-24 无锡天气预报`

### 5.3 Prompt 中只消费上下文，不直接发明上下文

正确做法是：

1. 先在编排层生成“时间上下文（北京时间）”
2. 再把该上下文以稳定格式传入 prompt 模板
3. 模型按模板规则消费上下文，而不是自己猜“今天是哪天”

这能显著降低“模型谨慎过度”或“模型乱猜时间”的概率。

## 6. 提示词模板化设计原则

### 6.1 模板文件放到 `resources/prompts`

当前实现采用：

```text
rag-admin-server/src/main/resources/prompts/ai/chat/
```

这样做的好处：

- prompt 与业务代码解耦
- 便于 review 时单独审阅模板文本变化
- 便于未来做 A/B、灰度、版本回滚
- 避免 Service 类里塞满多行字符串

### 6.2 模板文件统一使用 `.st`

当前约定模板后缀为 `.st`，表示纯文本模板文件。

现阶段不引入完整模板引擎，只保留最小能力：

- `${variable}` 占位

保持足够简单，避免为了几处变量替换引入额外依赖和复杂语法。

### 6.3 通过 `@Value` 注入 `Resource`

不建议把大段提示词作为 Java 字符串常量继续硬编码。

当前推荐方式：

```java
@Value("classpath:prompts/ai/chat/app-general-system.st")
private Resource generalSystemPromptTemplate;
```

这样可以明确：

- 模板来源
- 模板文件路径
- 模板与当前业务类的依赖关系

### 6.4 统一由 `PromptTemplateService` 渲染

不要让每个业务类自己读文件、自己替换变量、自己处理异常。

当前统一使用：

- `PromptTemplateService.load(...)`
- `PromptTemplateService.render(..., Map<String, String>)`

它负责：

- 读取 classpath 模板
- 做 UTF-8 读取
- 做缓存
- 做 `${variable}` 替换
- 在缺变量时快速失败

### 6.5 模板只承载文本，不承载复杂流程控制

现阶段不推荐把大量 `if / else / loop` 逻辑塞进模板语法。

更合理的做法是：

- 业务代码负责分支决策
- 模板只负责每个分支的最终文本表达

例如：

- “知识片段 + 联网摘要”
- “只有知识片段”
- “只有联网摘要”
- “两者都没有”

这些分支由业务代码判断，然后渲染不同模板即可。

本次已将以下提示词收口到模板文件：

- 前台问答 `AppChatService`

## 8. 测试策略经验

### 8.1 单测不要依赖 Spring 容器隐式注入

当前很多单测是 Mockito 级别的纯单元测试，不会启动 Spring。

因此通过 `@Value` 注入的 `Resource` 在测试里不会自动赋值。

解决方式：

- 在测试中用 `ClassPathResource` 显式注入模板资源
- 使用 `ReflectionTestUtils.setField(...)` 补齐模板字段

这样既保留生产代码的 `@Value` 形式，也不会把单测强行升级为 `@SpringBootTest`

### 8.2 模板渲染要缺变量即失败

如果模板里有 `${question}`，而业务代码没传，就不应该静默吞掉。

实践建议：

- 渲染阶段缺变量直接报错
- 尽早在单测或本地联调阶段暴露模板与代码不同步的问题

## 9. 什么时候仍然要上专用 tool / MCP

模板化和 prompt 优化不能替代专用能力。

以下场景仍应优先评估专用 tool / MCP：

- 天气
- 股价 / 汇率
- 地图 / 距离 / POI
- 航班 / 火车 / 物流
- 法规 / 政策生效时间
- 企业内部审批状态、任务状态、监控状态

原因很简单：

- 这些问题对结构化、确定性、时效性要求更高
- 通用 web search 只是弱结构化证据，不是强事实源

因此本项目的长期路线应是：

1. 优先做输入归一化
2. 优先做能力路由
3. prompt 模板只负责消费上下文和控制表达
4. 对强约束领域逐步接入专用 tool / MCP

## 10. 后续演进建议

### 10.1 继续把“现实世界上下文”结构化

后续可考虑把以下信息从文本描述升级为结构化对象：

- `currentDate`
- `timezone`
- `location`
- `recencyRequired`
- `userIntentCategory`

这样编排层与 tool 路由会更稳定。

### 10.2 为高价值模板增加版本管理

后续可考虑：

- 模板版本号
- 模板变更记录
- 针对不同模型的模板变体

但当前阶段先保持最小实现即可，不必过早做复杂 prompt 平台。

### 10.3 避免模板散落到业务域外的任意位置

统一把 AI prompt 模板收敛到 `resources/prompts`，不要出现：

- 一部分在 Java 常量里
- 一部分在 `docs/`
- 一部分在前端常量里
- 一部分在数据库里

否则维护成本会快速失控。

## 11. 最终结论

本次经验可以收敛为五句话：

- 现实世界问答的稳定性，不应只靠补 prompt
- 相对时间归一化属于通用上下文能力，不是天气特判
- 能走专用 tool / MCP 的问题，不应长期依赖通用联网摘要
- 大段提示词必须模板化、资源化，而不是继续硬编码在 Service 中
- Prompt 的职责是消费上下文和约束表达，不是替代输入标准化与能力路由

以上原则应作为 `ragAdmin` 后续扩展现实世界问答、联网增强和多模型编排时的默认约束。
