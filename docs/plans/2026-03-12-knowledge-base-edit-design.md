# 知识库编辑页设计说明

## 1. 目标

本次实现目标是在现有知识库创建页基础上，补齐“知识库编辑”最小闭环：

- 从知识库列表进入编辑页
- 编辑页加载知识库详情并回填表单
- 提交更新后返回列表页
- 列表页自动刷新并提示更新成功

本轮重点是复用现有表单组件和交互模式，避免为编辑场景重复造轮子。

## 2. 方案选择

本次采用“独立编辑页 + 复用现有表单组件”的方案。

选择原因如下：

- 保持创建页和编辑页职责清晰，不把两种场景强行塞进一个页面容器
- 复用 `KnowledgeBaseForm`，减少重复字段定义、校验逻辑和样式维护
- 后续如果再补详情页、文档入口、启停操作，扩展边界更清楚

路由设计如下：

- 创建页：`/knowledge-bases/create`
- 编辑页：`/knowledge-bases/:id/edit`

## 3. 页面与交互设计

编辑页继续使用知识库表单组件，但允许通过参数切换以下文案：

- 页面标题
- 页面说明
- 提交按钮文案

列表页增加“编辑”操作入口。交互流如下：

1. 用户从知识库列表点击“编辑”
2. 页面根据知识库 ID 加载详情
3. 将详情映射到表单结构
4. 用户修改并提交
5. 更新成功后返回列表页
6. 列表页识别一次性成功标记并刷新、提示

## 4. 数据流设计

### 4.1 详情加载

编辑页通过 `GET /api/admin/knowledge-bases/{kbId}` 加载详情。

详情成功后：

- 回填知识库表单
- 进入可编辑状态

详情失败时：

- 显示明确错误提示
- 保留返回列表入口
- 不展示误导性的空表单

### 4.2 模型选项加载

模型加载策略与创建页保持一致：

- 优先请求真实模型列表
- 按聊天模型 / Embedding 模型拆分
- 请求失败时退化为平台默认模型兜底

编辑页需要额外满足一个要求：

- 如果知识库详情中已绑定模型，但模型列表接口失败或未返回对应项，不能在页面层把已有绑定静默清空

因此编辑页应以详情数据为准保留当前值，再由模型列表补充展示能力。

### 4.3 保存更新

前端新增更新接口封装，对接 `PUT /api/admin/knowledge-bases/{kbId}`。

提交流程：

1. 用户修改表单
2. 前端执行基础校验
3. 调用更新接口
4. 成功后跳回列表页，并携带一次性 `updated=1` 标记
5. 列表页识别标记后刷新并提示“知识库更新成功”
6. 刷新后清除标记，避免重复提示

## 5. 复用边界

本轮只做必要复用，不额外引入全局状态或复杂抽象。

建议复用点如下：

- `KnowledgeBaseForm`：复用字段渲染、校验和样式
- 模型选项筛选逻辑：从创建页提取为可复用工具
- 表单归一化逻辑：创建 / 编辑共用

不在本轮引入：

- 创建 / 编辑共用单一页面容器
- 全局模型选项缓存
- 未保存离开提示
- 字段级脏值追踪

## 6. 错误处理

错误处理规则如下：

- 表单校验失败：字段级提示，不允许提交
- 详情加载失败：页面级错误提示，并允许返回
- 更新失败：页面消息提示，保留用户输入
- 提交中按钮进入 loading，防止重复提交

## 7. 验证范围

本轮至少完成以下验证：

- 列表页可进入编辑页
- 编辑页可成功加载详情并回填表单
- 模型接口失败时不误清空已有模型绑定
- 更新成功后可回到列表页并自动刷新、提示
- 前端项目可通过 `npm run build`

## 8. 后续建议

在编辑闭环完成后，建议继续按以下顺序推进：

1. 知识库详情页
2. 知识库启停操作
3. 知识库文档列表与上传入口
