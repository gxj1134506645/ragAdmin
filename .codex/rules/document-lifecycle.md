# 文档生命周期规则

## 目标

让 `docs/architectures`、`docs/plans`、`docs/tasks` 保持轻量、单一事实源和可持续收口，避免随着工程推进持续堆积过期文档。

## 三类文档职责

- `docs/architectures`
  - 当前正式生效的架构与技术设计
  - 属于稳定事实源
- `docs/plans`
  - 当前仍有效的阶段性方案、设计推演与实施计划
- `docs/tasks`
  - 当前执行型文档，例如任务拆解、联调步骤、验收清单

## 默认流转关系

1. 先由 `architectures` 定义稳定边界与长期约束
2. 再由 `plans` 形成某个阶段或大功能的落地方案
3. 再由 `tasks` 拆成 Agent 可稳定执行的动作
4. 执行后形成的长期有效结论，再回写到 `architectures`

## 命名规则

- 文件名优先使用稳定英文 slug
- 不使用按年月日追加的文件名前缀
- 文件名应表达主题，而不是表达创建当天时间

推荐示例：

- `system-architecture.md`
- `chat-memory-plan.md`
- `api-acceptance.md`
- `frontend-phase-1-tasks.md`

## 何时检查 `plans` 与 `tasks`

满足以下任一条件时，应检查是否需要收口：

- 一个大功能完成后
- 一次阶段性开发结束后
- 准备提交较大阶段成果前
- 路线发生明显变更后

## `plans` 保留条件

满足以下任一条件时保留：

- 仍在推进
- 仍是当前有效方案
- 后续实现仍会持续参考

## `plans` 删除条件

满足以下全部条件时可删除：

- 计划已完成，或已被新方案替代
- 文档不再继续指导后续实现
- 长期有效结论已回写到 `docs/architectures`

## `tasks` 保留条件

满足以下任一条件时保留：

- 仍在执行
- 近期仍会复用
- 当前仍是唯一有效的联调、验收或排查依据

## `tasks` 删除条件

满足以下全部条件时可删除：

- 任务已完成
- 不再作为近期联调、验收或排查依据
- 内容没有明显复用价值

## 回写规则

满足以下任一条件时，应优先回写到 `docs/architectures`：

- 结论会长期约束实现边界
- 结论会影响 API、Schema、模块职责或技术选型
- 结论适用于多个后续功能，而不是一次性任务

## 禁止事项

- 同一主题长期并存多个过期 plan
- 让 `plans` 承担长期事实源职责
- 让 `tasks` 演变成无限追加的执行流水账
- 为保留历史痕迹长期堆积已失效文档
