# 工程记忆写入规则

## 目标

让运行目录下的 `memory` 只沉淀高价值、可复用、能提升后续成功率的结论。

## 适合写入 memory 的情况

满足以下任意两项时，优先考虑写入 memory：
- 未来有较高重复概率
- 会影响边界、架构、契约或质量
- 已形成明确纠正动作
- 结论可复用于多个后续任务
- 经过验证，适合作为稳定经验保留

## 更适合留在任务上下文中的情况

以下情况更适合作为当前任务记录，不进入 memory：
- 一次性偶发问题
- 纯环境噪音
- 还在讨论中的猜测
- 过程性的普通尝试记录

## 写入目标选择

- `corrections.md`
  - 已发生问题，并已形成纠正结论
- `observations.md`
  - 已识别出有价值观察，后续可继续验证
- `learned-rules.md`
  - 已验证稳定，可跨任务复用
- `anti-patterns.md`
  - 已识别出高风险模式，并有更优替代方式
- `evolution-log.md`
  - 运行机制、目录结构、角色分工发生演进

## 维护方式

- 同类问题优先更新已有条目
- 每条记录优先补齐场景、原因、动作、适用范围
- 每个阶段可对过期内容做一次收敛整理
- 大块功能模块完成、阶段性提交、路线变化或重大验收通过后，主动更新 `project-progress.md`
- `project-progress.md` 的宏观阶段、已完成主线、进行中重点或未启动重点发生变化时，同步更新 `session-brief.md`
- 简单修复、单点配置核查、一次性排查默认不更新进度记忆，除非改变了项目阶段或长期执行规则
